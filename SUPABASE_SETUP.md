# Supabase 설정 가이드

프론트에서 Supabase를 쓰려면 아래를 한 번만 설정하면 됩니다.

---

## 1. config.js 값 넣기

- **SUPABASE_URL**: 대시보드 > Project Settings > API > Project URL
- **SUPABASE_ANON_KEY**: 같은 화면의 anon public key
- **ADMIN_PASSWORD**: 관리자 화면(admin.html) 로그인 비밀번호 (원하는 값으로 변경)

---

## 2. Storage 버킷 (서류 업로드용)

- **config.js**의 **`STORAGE_BUCKET`** 값이 Supabase Storage에 만든 버킷 이름과 **완전히 같아야** 합니다.
- 대시보드 **Storage**에서 버킷 이름을 **`reservation-files`** 로 만들었으면 config도 `'reservation-files'` 로 두면 됩니다. (다른 이름으로 만들었다면 config를 그 이름으로 수정)
- **Public bucket**으로 두면 업로드된 서류의 공개 URL로 관리자 화면에서 링크 클릭 시 바로 조회할 수 있습니다.
- **anon 업로드(INSERT)** 정책이 필요합니다. (`supabase_rls_policies.sql` 참고)

### "Bucket not found" (404) 나올 때

관리자 페이지에서 서류 링크를 눌렀을 때 **`{"error":"Bucket not found"}`** 가 나오면:

1. Supabase 대시보드 → **Storage** 로 이동합니다.
2. 버킷 목록에 **`reservation-files`** (또는 config에 넣은 이름)가 있는지 확인합니다.
3. **없다면** → **New bucket** 으로 이름을 정확히 **`reservation-files`** 로 만들어 줍니다. (공백·대소문자 구분)
4. **다른 이름으로 이미 만들었다면** → **config.js** 의 `STORAGE_BUCKET` 을 그 이름과 똑같이 수정합니다.

---

## 3. RLS (reservations 테이블)

예약 테이블에 RLS를 켜고, **익명(anon)으로 예약 INSERT**와 **전체 SELECT**(관리자 화면용)를 허용합니다.

```sql
-- RLS 활성화
alter table public.reservations enable row level security;

-- 누구나 예약 1건 추가 가능 (사용자 화면)
create policy "Allow anon insert"
on public.reservations for insert to anon
with check (true);

-- 조회는 모두 허용 (관리자 화면에서 anon key로 조회 가능, 실제 접근은 admin 비밀번호로 제한)
create policy "Allow anon select"
on public.reservations for select to anon
using (true);
```

나중에 관리자만 조회하도록 바꾸려면 Supabase Auth로 관리자 로그인 후, `auth.uid()`가 관리자일 때만 SELECT 허용하는 정책으로 교체하면 됩니다.

### 3-2. 한 번에 적용 (권장)

**서류 업로드 오류: new row violates row-level security policy** 가 나면 아래 SQL을 한 번만 실행하면 됩니다.

프로젝트 루트에 **`supabase_rls_policies.sql`** 파일을 두었습니다.  
Supabase 대시보드 → **SQL Editor** → 새 쿼리에서 해당 파일 내용을 붙여넣고 **Run** 하세요.

- `reservations` 테이블: anon **INSERT** / **SELECT** 허용  
- `storage.objects` (reservation-files 버킷): anon **INSERT** / **SELECT** 허용  

이미 같은 이름의 정책이 있으면 `drop policy if exists` 후 다시 생성하므로, 그대로 전체 실행해도 됩니다.

## 4. 테이블 스키마 (실제 사용 중)

`reservations` 테이블은 아래와 같습니다. 코드가 이 컬럼명에 맞춰져 있습니다.

| 컬럼            | 타입      | 비고 |
|-----------------|-----------|------|
| id              | bigint    | PK, generated by default as identity |
| created_at      | timestamptz | default now() |
| reservation_date | date     | 예약일 |
| reservation_time | text     | 예: "08:30" |
| company_name    | text     | 업체명 |
| car_number_1    | text     | 차량번호1 |
| car_number_2    | text     | nullable |
| material_info   | text     | 자재 정보 |
| contact         | text     | 연락처 |
| doc_url_1       | text     | 서류 URL 1 |
| doc_url_2       | text     | 서류 URL 2 |
| doc_url_3       | text     | 서류 URL 3 |
| status          | text     | '대기' / '승인' / '반려' |
| person_info     | jsonb    | 출입 인원 등, 예: `{"text": "총 2명 (홍길동, 이철수)"}` |

---

## 5. 로컬에서 확인

`config.js`에 URL과 anon key를 넣은 뒤, `index.html`을 브라우저에서 직접 열어서(또는 로컬 서버로) 날짜 선택 → 시간 슬롯 조회, 예약 제출이 되는지 확인하면 됩니다.  
서류 업로드 시 Storage 버킷 **`reservation-files`** 와 anon 업로드 정책이 없으면 업로드에서 에러가 나므로, 먼저 Storage 설정을 마치는 것이 좋습니다.

---

## 참고

Cursor에서 Supabase MCP로 연동해 두었다면, 대시보드/DB 조회는 MCP를 통해 할 수 있습니다. 이 코드베이스에서는 당신의 Supabase 프로젝트에 직접 접속해 버킷을 확인하는 기능은 없습니다.
